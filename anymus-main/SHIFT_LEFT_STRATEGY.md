# ANYMUS — Shift Left Strategy

## What is Shift Left Testing?

Shift Left means moving quality activities as early as possible in the software development lifecycle — ideally starting at the requirements phase, not after development ends. The earlier a defect is caught, the cheaper and faster it is to fix.

Traditional QA waits for a build. ANYMUS doesn't.

## The Five Shift Left Principles in ANYMUS

### 1. Requirement-Traced Test Generation
Every test case in `test-cases/ui-manual-tests.md` and `test-cases/api-manual-tests.md` was generated by an AI agent from the live InvenTree documentation — before any automation was written. Test IDs trace back to documented business rules and schema constraints.

This is testing at the **requirements phase**, not the testing phase.

### 2. Contract-First API Validation (schemathesis)
Before writing a single hand-crafted API test, `schemathesis` auto-generates tests from the InvenTree OpenAPI spec. This validates the API contract itself — catching regressions in required fields, response structures, and status codes that a hand-written test suite might miss.

If the contract breaks, we know before the UI team writes a single selector.

### 3. Infrastructure-Resilient Testing (Mock Mode — D2)
Traditional test pipelines fail when the server is down. ANYMUS uses the `responses` library to record real API responses as JSON fixtures. With `MOCK_MODE=true`, the full API test suite runs against these fixtures — no InvenTree instance required.

This enables: offline development, fast CI feedback, and isolation of test logic from infrastructure reliability.

### 4. Pre-flight Health Gate (D3)
Before any test suite runs — API or UI — ANYMUS pings all InvenTree Part API endpoints and reports their status in a colored table. The gate works at two levels:

- **API tests**: `test_meta_health.py` runs first; skips the suite if endpoints are unreachable
- **UI tests**: `api_health_gate` session fixture in `conftest.py` skips all UI tests if the API is down — preventing minutes of browser automation against a broken backend from consuming CI/CD runner time

This is Shift Left applied to *infrastructure quality* — fail fast, fail cheap.

### 5. Visual Regression as a Safety Net (D1)
DOM-based assertions validate behaviour. Screenshot-based assertions validate *appearance*. ANYMUS captures pixel-perfect baselines of critical Part pages using Playwright's built-in visual comparison. Any CSS regression, layout shift, or unintended UI change fails a visual test — even if every DOM assertion passes.

Visual tests are not a replacement for functional tests. They are an additional left-shifted quality layer that catches what functional tests cannot.

## Why This Matters for the Jury

| Jury Criterion | ANYMUS Evidence |
|---|---|
| **Originality** | Health check as a test gate is novel; mock mode + visual regression together is uncommon in a single pipeline |
| **Execution** | All five principles are implemented as runnable artefacts — not slide concepts |
| **Technology** | schemathesis + responses + Playwright visual + GitHub Actions CI — modern, non-trivial stack |
| **Usability** | `MOCK_MODE=true` for offline use; `UPDATE_SNAPSHOTS=true` for baseline refresh; health check standalone CLI |
| **Scalability** | Parameterized tests, data-driven fixtures, mock layer — the pipeline scales to new endpoints with minimal effort |
